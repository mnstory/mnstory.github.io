Python多进程并行函数对比

发表于 2017-09-15 |

Python因为解释器本身实现的比较挫（是的，就是代码写得挫），有一个GLI(Global Interpreter Lock)锁，导致多线程基本上是一个人装出来的几个分身，所以如果要并行，推荐使用多进程，有multiprocessing库，封装的还算简洁，但是毫无疑问，多进程通信成本会比多线程高一些。但是，有什么办法？用Python你还在乎性能？

别人在不在乎，我是不知道，反正最近我们有个模块是非常在乎，用Python是因为历史原因和现实原因，有本事换成Go，你得问问那一大堆支撑库答应不答应。

Python多进程并行，可调用map和apply，两个函数都有对应的ansy版本，ansy版本返回的是Result对象，此对象可用于后面等待结果。

并行功能比较适合那些执行之间不存在时间依赖(可能后传人的会先执行)或者结果依赖(后面的执行依赖前面的输出状态)的模块。

异步功能比较适合后续代码的执行不需要依赖当前步骤执行结果的模块，比如插入一条数据到数据库，插入过程比较耗时，可以异步执行，我的主进程可以继续执行处理构造下一条数据的工作。

  1. apply完全是阻塞模式一个一个调用，利用不上多进程的优势，所以耗时是串联耗时，为15s。
  2. map也是阻塞，但是，内部是并行的，所以能利用上多进程优势，但是同时主进程会被阻塞住，并行耗时9s，和apply_async/map_async运行时长一致。
  3. apply_async，map_async是非阻塞的，主进程可以立刻退出，直接到后面使用res.get()等待结果，在最后一条apply_async/map_async到第一条res.get()之间，主进程完全可以估摸着时间做其他事情。这两种方法是实现异步的最佳方法，但是map_async只接受一个参数(当然你可以用tuple将参数包裹，但得修改被调用的函数原型)。

结果回调处理

在异步调用的时候，可以传入回调函数，不需要自己调用res.get()，当然，传入回调后，你还是可以调用res.get()，两者并不冲突。传入的回调函数，会在result ready后自动调用。
apply_ansy因为是一个一个函数调用的，所以回调的时候，也是一个一个的结果传入回调函数，而map_async是打包的结果（所以，如果处理结果比较耗时，建议使用apply_ansy，这样结果会分先后传入，让你提前处理，如果不耗时，可以使用map_async，一次统一处理相对逻辑更简洁）。示例代码：

  1. 主进程的主线程4348/MainThread不到1秒就将所有参数apply给了线程池，然后就调用sleep等待。
  2. 等待线程池调用完成一个(apply_async)或所有(map_async)队列后，在主进程里新启动一个线程Thread-3去执行result callback。
  3. 如果被调用函数work内部出现异常，result callback和res.get()方式表现会有差异：
    • 对于apply_async，result callback方式是忽略掉出问题的那一次调用，而其他没抛异常的work执行结束后，会正常调用result callback。
    • 对于map_async，只要有一个work执行抛出异常，会导致整个result callback不被调用。
    • 不论是apply_async还是map_async，如果用res.get()方式，work内部抛出的异常会传导到此。
